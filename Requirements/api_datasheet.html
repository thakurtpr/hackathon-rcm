Backend API Datasheet

Project: AI Student Loan & Scholarship System Owner: Person B — Backend Engineer Version: v1.0 Base URL: https://api.loanai.dev/v1
All endpoints require Authorization: Bearer <JWT> header unless marked [PUBLIC]. Timestamps are ISO 8601 UTC. All IDs are UUIDs. Responses always include { "status": "success"|"error", "data": {}, "message": "" }

MODULE 3 — DOCUMENT SERVICE

Handles document upload to MinIO and triggers AI processing via Kafka. DB: PostgreSQL documents table + MinIO object storage.

POST /documents/upload

Multipart upload. Stores encrypted file in MinIO, creates document record, fires document.uploaded Kafka event for Person C's OCR pipeline.

FieldTypeRequiredNotes
user_idUUIDREQ
doc_typeenumREQaadhaar|pan|selfie|marksheet|income_cert|bank_passbook|admission_letter|caste_cert
filebinaryREQMax 10MB. PDF or JPG/PNG

Response 201: { doc_id, minio_path, status: "processing", upload_timestamp }

After upload, fire Kafka event: document.uploaded with { doc_id, user_id, doc_type, minio_path }. Person C's OCR service listens to this topic.

GET /documents/{user_id}/status

Returns verification status of all documents for a user.

Response: { docs: [{ doc_type, doc_id, status: "pending"|"verified"|"rejected", rejection_reason }] }

MODULE 5 — ELIGIBILITY & SCORING SERVICE

Receives sub-scores from AI services (via Kafka), computes weighted composite score, writes result. DB: PostgreSQL eligibility_scores table.

POST /eligibility/compute

Called internally when all sub-scores are ready. Computes final composite score and fires eligibility.calculated event.

FieldTypeRequiredNotes
app_idUUIDREQ
academic_scorefloat (0–100)REQFrom eligibility AI svc
financial_scorefloat (0–100)REQFrom eligibility AI svc
pq_scorefloat (0–100)REQFrom Person C behavioral svc
doc_trust_scorefloat (0–100)REQFrom Person C OCR/face svc
kyc_completenessfloat (0–100)REQComputed from profile completeness
fraud_flagbooleanREQIf true, auto-reject regardless of score

Weights applied server-side: Academic×0.25 + Financial×0.30 + PQ×0.20 + DocTrust×0.15 + KYC×0.10

Response 200: { composite_score, band: "approved"|"review"|"rejected", approved_amount, interest_rate_band, pq_override_applied: bool }

PQ override rule: if composite 50–69 AND pq_score >= 80, upgrade band to "approved". Record pq_override_applied: true in audit log.

GET /eligibility/{app_id}

Returns full score breakdown. Used by frontend explainability panel and by admin dashboard.

Response: All sub-scores + composite + band + improvement_hints array (generated by Person C explainability svc)

Integration checklist for Person B:
1. Set up Docker Compose (Postgres + Redis + Kafka + MinIO + Qdrant) on Day 1 and commit it.
2. Share OpenAPI YAML with Person A and C before writing any endpoint code.
3. Kafka consumer groups: use backend-svc as group ID for all B consumers, ai-svc for C.
4. All PII fields (aadhaar, pan, bank account) must be AES-256 encrypted at rest — never store plaintext.
5. Audit log every state transition — append-only, never update or delete audit rows.
6. Person C calls /ai/*-result endpoints; Person B never calls Person C's AI endpoints directly except for /ai/behavioral/questions.
