Screen 5 — Behavioral Assessment

8 questions, one per screen. Mix of MCQ and free text. Timed. Clean card UI — not a form.

Assessment UI spec

  • Full-screen card layout. Question text large (18px). No distractions.
  • Top bar: question counter ("Question 3 of 8") + progress bar
  • Timer shown only for free-text questions (2 min per question). Hide timer for MCQ.
  • MCQ: 4 option cards. Click to select, highlight selected. Next button activates on selection.
  • Free text: large textarea, character count shown (min 80 chars to enable Next)
  • No back navigation — once submitted a question cannot be changed (prevents gaming)
  • On final question submit → loading screen "AI is evaluating your responses..." with animated progress (fake 8-second wait to set expectation)

Screen 6 — Live Application Status Tracker

The most technically interesting screen. Real-time pipeline visualization driven by WebSocket.

Pipeline stages to display

StageStatus statesIcon/colour
Application submitteddoneGreen check
Documents verifiedpending / done / failedSpinner / check / red X
Behavioral assessmentpending / doneSpinner / check
Fraud checkpending / done / flaggedSpinner / check / warning
Eligibility scoringpending / doneSpinner / check
Final decisionpending / approved / rejected / reviewSpinner / green / red / amber

WebSocket connection

  • Connect to WS /applications/{app_id}/live on screen mount using socket.io-client
  • Listen for state_change event → update Zustand useApplicationStore → re-render stage list
  • On WebSocket disconnect: auto-reconnect with exponential backoff (1s, 2s, 4s, 8s)
  • If WS unavailable: fall back to polling GET /applications/{app_id}/status every 5 seconds
  • Show "Live" green dot in top corner when WS is connected, grey dot when polling
Estimated time display: show "Usually takes 3–5 minutes" on first load. Update to "Almost done..." when fraud_check stage completes.

Screen 7 — Result + Explainability Screen

Shown after final decision. The most emotionally important screen — design it carefully.

Layout — approved case

  • Large green hero banner: "Congratulations! Your loan has been approved."
  • Approved amount, interest rate band, and first disbursal date prominently shown
  • Score breakdown radar chart (recharts RadarChart) — 5 dimensions visualized
  • PQ badge prominently shown: "Potential Quotient: 84 / 100"
  • If PQ override was applied: show special badge "Approved by Potential — your drive made the difference"
  • Semester disbursal schedule table — all semesters with dates and amounts
  • Matched scholarships section below — cards with scholarship name, amount, apply button

Layout — rejected case

  • Amber (not red) banner: "Your application needs more time." — avoid harsh language
  • Same score breakdown radar chart
  • Improvement hints section — up to 3 cards, each with a specific action ("Add a co-applicant to boost your financial score by ~18 points")
  • Re-application date shown ("You can re-apply after 90 days")
  • Scholarship suggestions even on rejection — they may still qualify for scholarships

SECTION 5 — API LAYER

Axios setup (src/api/client.ts)

  • Base URL from VITE_API_BASE_URL env variable
  • Request interceptor: attach Authorization: Bearer {token} from useAuthStore
  • Response interceptor: on 401 → call POST /auth/refresh → retry original request once → if refresh fails, logout + redirect to /login
  • Response interceptor: on 500 → show generic toast "Something went wrong. Please try again."
  • Never expose raw axios errors to the UI — always transform to { message, code }
