Person C — AI / LLM Engineering Sheet

Project: AI Student Loan & Scholarship System Owner: Person C — AI/LLM Engineer Stack: Python 3.11 + FastAPI + LangGraph + Groq API Service base: http://localhost:8001
Person C runs a completely separate Python FastAPI service. Person B calls C's question endpoint directly. For all other results (OCR, fraud, PQ scores, scholarships, explanations), Person C posts back to Person B's bridge endpoints. Person C listens to Kafka for triggers. Never touches PostgreSQL directly — always reads student data via Person B's REST API.

SECTION 5 — PIPELINE SPECS

Pipeline 1 — OCR + document parsing

1
Receive document.uploaded Kafka event. Extract minio_path and doc_type.
2
Fetch file bytes from MinIO using Python SDK: minio_client.get_object(bucket, minio_path)
3
If PDF: convert to images using pdf2image.convert_from_bytes()
4
Run PaddleOCR: ocr.ocr(img_array, cls=True) → extract raw text blocks
5
Run LayoutLM or prompt Groq with raw text to extract structured fields per doc_type (see schema below)
6
Compute doc_trust_score: OCR confidence avg × tamper_check × completeness_score
7
POST result to Person B: POST /ai/kyc-result

Extracted fields per doc type

doc_typeFields to extract
aadhaarname, dob, gender, aadhaar_last4, address_state
panname, pan_number, dob
marksheetstudent_name, institution, year, percentage_or_cgpa, pass_fail
income_certholder_name, annual_income, issuing_authority, issue_date
bank_passbookaccount_holder_name, account_number_last4, bank_name, ifsc
semester_marksheetstudent_name, semester_number, sgpa, result (pass/fail)

Pipeline 2 — Face match

1
Triggered when both aadhaar and selfie docs are uploaded for the same user.
2
Fetch both images from MinIO.
3
Detect face in each image using InsightFace: app.get(img) → get 512-dim embedding.
4
Compute cosine similarity between the two embeddings.
5
Threshold: score ≥ 0.85 → face_match_pass: true. Score 0.70–0.84 → flag for manual review. Below 0.70 → fail.
6
Also store face embedding in Qdrant pool for fraud dedup check.

Pipeline 3 — Behavioral PQ engine

Question generation prompt logic

Prompt sent to Groq (Llama 3 70B) includes: student's course, institution tier, home state, family income band, last exam score. LLM generates 8 contextual questions. Mix must be: 3 MCQ situational + 2 MCQ financial literacy + 2 free-text goal/resilience + 1 free-text initiative.
DimensionWeight in PQQuestion typeWhat it measures
Financial responsibility20%MCQ situationalWill they prioritise repayment?
Resilience20%Free textDropout probability
Goal clarity20%Free textEmployability post-graduation
Risk awareness15%MCQ financial literacyDo they understand loans?
Initiative / agency15%MCQ situationalSelf-driven income trajectory
Social capital10%MCQ situationalAccess to job opportunities

Scoring logic

  • MCQ: map each option to a score (0, 33, 66, 100) based on rubric defined in app/prompts/mcq_rubric.json
  • Free text: send answer to Groq with scoring rubric prompt → extract integer score 0–100 from response
  • Run a rule-based sanity check: if LLM returns score outside 0–100 or non-numeric, re-run once then default to 50
  • PQ = weighted sum of 6 dimension scores using weights above
  • Flag if total time_taken_seconds across all questions < 60 — possible gaming

Pipeline 4 — Fraud detection

CheckHow to implementFlag condition
Duplicate PANCall GET /users/check-pan?pan= on Person B APISame PAN exists in another application
Duplicate AadhaarCall GET /users/check-aadhaar?hash= on Person B APISame Aadhaar hash in another application
Face pool dedupSearch Qdrant face embeddings collection with new embedding, threshold 0.92Match found in pool = same person different name
Income inconsistencyCompare OCR-extracted income vs stated family_annual_income in profileDifference > 40%
Doc metadata tamperCheck PDF creation date vs modification date using PyPDF2 metadataModified after creation by > 1 day
Velocity checkCall Person B API — how many apps from same mobile/email in last 30 daysMore than 2 applications

fraud_confidence = (num_flags / total_checks) × 100. If any single check is a hard fail (duplicate PAN/face), set fraud_flag = true regardless of confidence.

Pipeline 5 — Scholarship matcher (Qdrant RAG)

1
On startup, embed all scholarships from data/scholarships/ JSON into Qdrant collection scholarships using sentence-transformers/all-MiniLM-L6-v2. Run once, skip if collection already populated.
2
For each application: fetch student profile from Person B. Build profile text: "Student: [course] at [institution], [state], [category], income [band], CGPA [score]"
3
Embed profile text using same model. Query Qdrant: client.search(collection_name="scholarships", query_vector=embedding, limit=10)
4
Filter results: deadline must be in future, student must meet category/income/score criteria. Take top 5 after filtering.
5
For each match, generate 1-sentence reason using Groq: "You qualify because your CGPA and SC category match this merit-cum-means scheme."
6
POST to Person B: POST /ai/scholarship-result

Pipeline 6 — LangGraph agent orchestrator

Triggered by eligibility.calculated Kafka event. Runs 5 key agents in sequence using LangGraph StateGraph.
AgentInputOutput
Student profile agentuser_id → fetch profileStructured profile summary for context
Document verification agentdoc statuses from Person BDoc trust verdict + any flags
Eligibility eval agentcomposite score + sub-scoresScore interpretation + band rationale
Policy compliance agentLoan amount + RBI rules (RAG)Policy pass/fail flags
Final approval agentAll above outputs combinedFinal decision rationale + improvement hints

Each agent is a LangGraph node. State is a TypedDict passed between nodes. Use add_conditional_edges for the fraud short-circuit (if fraud_flag = true, skip to rejection explanation).

SECTION 9 — HACKATHON SHORTCUTS

Use Groq free API instead of running Llama locally — 500 tokens/sec, zero GPU needed. Sign up at console.groq.com.
For OCR in the demo: test with a real Aadhaar PDF. PaddleOCR works out of the box — pip install paddlepaddle paddleocr. First run downloads models automatically.
Pre-seed Qdrant with 20–30 real scholarship entries (NSP, post-matric, merit-cum-means) before the demo. The matcher works much better with real data.
For the Rajan demo account: pre-run the behavioral pipeline with Rajan's profile and cache his PQ score (88) in Redis. This way the demo doesn't need to wait 15 seconds for Groq scoring live.
InsightFace requires a C++ build environment. Run pip install insightface onnxruntime and test face match on the demo machine at least 2 hours before presentation. It sometimes fails on fresh installs.
Integration checklist for Person C:
1. Confirm Person B has GET /users/check-pan and GET /users/check-aadhaar endpoints — these are needed for fraud checks.
2. Agree on the exact payload shape for all 5 POST result endpoints before coding.
3. Test Kafka consumer independently before connecting to full pipeline.
4. Seed Qdrant collections on first startup — write a seed.py script that runs once.
5. Keep all Groq prompt templates in app/prompts/ — never hardcode prompts in logic files.
